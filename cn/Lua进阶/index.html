5. Lua的内存管理方案

5.1 Lua 内存基础

Lua与Python的内存管理不同，Lua仍然是将值赋给变量的模式，例如：

输出分别是：hello, changed.这就说明仍然采用朴素的赋值方式。而不是Python中的动态赋值

由于lua中有内存回收机制，所以本质上说lua与C/C++中的内存泄漏是不一样的，lua中有GC，理论上说是不会有内存泄漏的。

当它进行GC的时候，会从根部开始扫描所有的对象，如果某个地方对这个对象还有引用，那么就不会把这个对象collect，所以lua中的内存泄漏是指已经没有使用，但是外部依然还存在引用的对象。

Lua的GC算法使用的所谓的“Mark And Sweep”算法，简单理解，这个算法GC分为两个阶段，一个是标记（mark）阶段，这一阶段将所有系统中的引用的对象都逐一标记，而在清理（Sweep）阶段，把在mark阶段没有被标记的数据删除。

优化方法

  • 尽量少使用全局变量，由于Lua默认是全局变量，所以很容易造成内存泄漏，尽量使用local限定。
  • 在可能内存泄漏的地方先调用collectgarbage(“count”)获取内存的最初使用，然后在函数调用后进行collectgarbage(“collect”)收集，并使用collectgarbage(“count”)检测两次内存使用的差。或者手动进行显示的垃圾回收collectgarbage("step")

5.2 Garbage Collection

Lua使用垃圾收集器它在后台会时不时地运行，收集和销毁Lua程序访问的死对象。包括表，用户数据，函数，线程，字符串等在内的所有对象都需要进行自动内存管理。

Lua 实现了一个增量标记-扫描收集器。 它使用这两个数字来控制垃圾收集循环： 垃圾收集器间歇率和垃圾收集器步进倍率。 这两个数字都使用百分数为单位，例如，100表示为1.

  • 垃圾收集器间歇率控制着收集器需要在开启新的循环前要等待多久。 增大这个值会减少收集器的积极性。
    • 当这个值比 100 小的时候（低于100%），收集器在开启新的循环前不会有等待。
    • 设置这个值为 200 （200%）就会让收集器等到总内存使用量达到之前的两倍时才开始新的循环。
  • 垃圾收集器步进倍率控制着收集器运作速度相对于内存分配速度的倍率。 增大这个值不仅会让收集器更加积极，还会增加每个增量步骤的长度。 不要把这个值设得小于 100 ， 那样的话收集器就工作的太慢了以至于永远都干不完一个循环。 默认值是 200 ，这表示收集器以内存分配的"两倍"速工作。

Lua 提供了以下函数**collectgarbage ([opt [, arg]])**用来控制自动内存管理:

  • collectgarbage(“collect”): 做一次完整的垃圾收集循环。通过**参数 opt ** （表示option）它提供了一组不同的功能：
  • collectgarbage(“count”): 以 KB为单位返回 Lua 使用的总内存数。 这个值有小数部分，所以只需要乘上 1024 就能得到 Lua 使用的准确字节数（除非溢出）。
  • collectgarbage(“restart”): 重启垃圾收集器的自动运行。
  • collectgarbage(“setpause”): 将 arg 设为收集器的 间歇率。 返回 间歇率 的前一个值。
  • collectgarbage(“setstepmul”): 返回 步进倍率 的前一个值。
  • collectgarbage(“step”): 单步运行垃圾收集器。 步长"大小"由 arg 控制。 传入 0 时，收集器步进（不可分割的）一步。 传入非 0 值， 收集器收集相当于 Lua 分配这些多（K 字节）内存的工作。 如果收集器结束一个循环将返回 true 。
  • collectgarbage(“stop”): 停止垃圾收集器的运行。 在调用重启前，收集器只会因显式的调用运行。

5.3 神奇的local关键字

如果要创建实例，在lua中不需要new关键字，直接像下面这样写即可：

1.sb = CS.System.Text.StringBuilder();

注：可以将C#类型缓存，这样既方便使用，也可以提高性能：

1.local strbuilder = CS.System.Text.StringBuilder;

2.sb = strbuilder();

在 Lua 中使用 local 关键字进行变量声明有助于提高性能，主要是因为 local 关键字限定了变量的作用域在局部范围内，从而减少了变量查找的开销。这在与 Unity 或其他环境集成时特别有用。具体到您提到的情况，理解这一点需要考虑以下几个方面：

  1. 局部变量的访问速度更快：在 Lua 中，局部变量（使用 local 声明的变量）比全局变量（没有使用 local 声明的变量）拥有更快的访问速度。这是因为局部变量存储在栈上，而全局变量存储在一个全局的表中。当访问局部变量时，Lua 可以直接通过栈进行访问，而访问全局变量则需要通过表查找，这个过程相对来说更慢。
  2. 减少全局环境的污染：使用 local 还有一个重要的好处是减少了对全局环境的污染。在 Lua 中，如果不使用 local 声明变量，那么这些变量默认是全局的，这可能会导致意外的冲突和错误。局部变量的作用域限定在它被声明的区块内，这有助于维护代码的清晰和可维护性。
  3. 提高性能：在您的例子中，将 CS.System.Text.StringBuilder 类缓存到一个局部变量中，意味着每次创建 StringBuilder 实例时，Lua 不需要再去全局环境中查找这个类。这样可以减少查找时间，特别是在频繁创建实例的情况下，这种差异会更加明显。

所以有需要大量用到某个 外部部件、或者需要高频访问某个变量的时候，最好先把他用local变量缓存一下子（作为函数赋值），然后再进行后续的使用。

9. 协程

Lua 协同程序(coroutine)与线程比较类似：拥有独立的堆栈，独立的局部变量，独立的指令指针，同时又与其它协同程序共享全局变量和其它大部分东西。

协同程序可以理解为一种特殊的线程，可以暂停和恢复其执行。也就是说，一个时间段内，只能运行一个协程，在之后会进行协程的切换。

方法 描述
coroutine.create() 创建 coroutine，返回 coroutine， 参数是一个函数，当和 resume 配合使用的时候就唤醒函数调用
coroutine.resume() 重启 coroutine，和 create 配合使用
coroutine.yield() 挂起 coroutine，将 coroutine 设置为挂起状态，这个和 resume 配合使用能有很多有用的效果
coroutine.status() 查看 coroutine 的状态 注：coroutine 的状态有三种：dead，suspended，running，具体什么时候有这样的状态请参考下面的程序
coroutine.wrap() 创建 coroutine，返回一个函数，一旦你调用这个函数，就进入 coroutine，和 create 功能重复
coroutine.running() 返回正在跑的 coroutine，一个 coroutine 就是一个线程，当使用running的时候，就是返回一个 coroutine 的线程号

如果这个协程有参数，那么传参是在resume的第二个参数开始进行传入的。

游标（Cursor）：

  • 当你在 Lua 中使用 conn:execute 执行一个 SQL 查询时，它返回一个游标对象。
  • 这个游标允许你逐行遍历 SQL 查询的结果。在我们的例子中，SELECT name FROM sqlite_master WHERE type='table' 这条 SQL 命令返回了数据库中所有表的名称列表，游标就是用来遍历这个列表的。
  1. Fetch 函数：fetch 函数用于从游标中检索数据。每次调用 fetch 都会从结果集中获取下一行数据，直到没有更多数据时返回 nil。这允许你逐行遍历查询结果。
    • 参数 {}与 "a"的意义：
      • {} 是一个空的 Lua 表（table），用于存放从当前行中检索出的数据。每次调用 fetch 时，这个表都会被填充当前行的数据。
      • "a" 是一个模式参数，告诉 fetch 函数以关联数组的形式返回数据。在这种模式下，你可以使用列名作为键来访问每一列的数据。
  2. 使用 fetch 函数：
    • 第一次调用 fetch 时，你传递一个空表 {} 和模式 "a"。这样做的目的是创建一个新的表来存放第一行的数据，并告诉 fetch 以关联数组的形式返回数据。
    • 在后续的调用中，你传递之前创建的表 table 和同样的模式 "a"。这样 fetch 会用下一行的数据填充这个表，直到没有更多的行时返回 nil。

当你使用 LuaJIT 的 -b 选项来编译 Lua 脚本时，生成的 .out 和 .obj 文件实际上都包含 LuaJIT 可以理解的字节码，但它们的格式和用途有所不同：

  1. .out 文件：这是一个包含 Lua 字节码的文件。这个字节码文件是为了被 LuaJIT 直接执行而生成的。你可以直接使用 LuaJIT 运行这个 .out 文件，就像运行一个普通的 Lua 脚本一样。这种格式主要用于分发或部署时隐藏原始源代码，或者加快代码加载时间。
  2. .obj 文件：这是一个更特殊的格式，通常被称为“对象文件”。这种文件格式通常用于嵌入字节码到 C 应用程序中。通过生成 .obj 文件，你可以将 Lua 代码以字节码的形式嵌入到一个 C 语言编写的程序中。然后，这个程序可以使用 LuaJIT 的 API 来加载和执行这些字节码。这种做法通常用于将 Lua 代码集成到更大的项目中，尤其是那些用 C 或其他低级语言编写的项目。

至高方法：

  • 使用luajit进行编译文件，输出为.obj文件
  • 然后使用高级语言进行调用，或者就是lua里面直接require就行。

Lua Optimize

1. Frequent & local variable

高频访问的对象，应先赋值给一个local变量再进行访问

这个高频访问的对象，不仅仅是变量、还更多的是函数、table等。

原因分析

非local局部变量，即UpValue的访问通常需要更多的指令操作（需要进入_G进行查询），如果对应的UpValue是个Table的话，还需要进入Table的复杂的数据查询流程。

而local变量的访问，只需要一条简单寄存器读取指令，所以能节省出数据查询带来的性能开销。

需要注意的是，这个是针对超高频访问的内容。而且优化效果也有限。

Appendix

1. _G

G 是Global的意思。

定义在所有函数外部的变量我们可以称之为全局变量(Global Variable)，它的作用域默认是整个程序。但Lua作为一种嵌入式语言，代码段(chunk)都是由宿主应用调用的，它自身都不知道会被嵌入到哪个应用程序中。为了解决这个问题，它并没有使用全局变量，而是通过table对全局变量进行模拟。我们可以认为Lua语言把所有的全局变量保存在一个称为全局环境(Global Environment)的普通表中。

3. math库

math.random ([m [, n]])

此函数是 ANSI C 提供的简单伪随机生成器函数 rand 的接口。当不带参数调用时，返回范围 [0,1) 内的统一伪随机实数。 当使用整数 m 调用时，math.random 返回范围 [1, m] 内的统一伪随机整数。 当使用两个整数 m 和 n 调用时，math.random 返回范围 [m, n] 内的统一伪随机整数。
