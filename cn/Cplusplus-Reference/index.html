1.1 Constexpr

static和constexpr都是C++中的修饰符，它们可以用于变量和函数。当它们一起使用时，它们的含义是合并的。

  • static关键字在这里的含义是该变量或者函数是在编译时分配空间的，而不是在运行时。在类的成员变量中，static关键字表明该变量是类的所有实例共享的，不属于任何单个实例。
  • constexpr关键字告诉编译器，这个变量或者函数的值可以在编译时计算出来。换句话说，它的值是常量，可以在编译时确定。

当static和constexpr一起使用时，static constexpr的含义是这个变量或者函数在编译时分配空间，并且它的值在编译时就可以计算出来。这在需要在编译时使用常量，或者在编译时进行某种计算时非常有用。

constexpr是一种编译型语句，并且编译器会把被constexpr修饰的函数视作内联函数。

const具有两种语义：

  • 常量
  • 只读（后面有关const指针会用到）

为了区别这两种语义，就有了constexpr，目的在于告诉编译器，这些东西在编译器就可以被计算。

**注意！**这里用词是“计算”，而不是“被算出来”，这是两码事。

算出来，是指能够有一个确切的数字生成。

计算，是指能够提前构造一个算术通道，在运行时直接走这个通道即可。

注意，获得在编译阶段计算出结果的能力，并不代表 constexpr 修饰的表达式一定会在程序编译阶段被执行，具体的计算时机还是编译器说了算。

constexpr 还可以用于修饰函数的返回值，这样的函数又称为“常量表达式函数”。

注意，constexpr 并非可以修改任意函数的返回值。换句话说，一个函数要想成为常量表达式函数，必须满足如下 4 个条件。

  1. 整个函数的函数体中，除了可以包含 using 指令、typedef 语句以及 static_assert 断言外，只能包含一条 return 返回语句。

  2. 该函数必须有返回值，即函数的返回值类型不能是 void。

  3. constexpr修饰的函数，会被编译器当做内联函数，所以最好在声明时就进行定义。

  4. return 返回的表达式必须是常量表达式。

从今以后养成习惯：const表示只读，而constexpr才表示常量。

显式实例化的概念：

显式实例化（Explicit Instantiation）是 C++ 中的一个概念，用于告诉编译器要为特定的模板进行实例化，并生成对应的函数定义。它可以在编译器在编译期间生成模板的具体实现，以便在链接时能够找到相应的定义。

C++ 中的模板是一种通用的代码结构，可以根据不同的类型参数生成不同的代码。当使用模板时，编译器通常会根据实际使用的类型来生成相应的代码。这被称为隐式实例化（Implicit Instantiation），它在需要时自动进行。

然而，对于某些模板，特别是那些只在特定的类型上使用的模板，编译器可能无法自动推断需要实例化的类型。在这种情况下，显式实例化提供了一种方式，让开发者明确指示编译器要为特定的模板进行实例化。

1.5 编译参数

CMake的-D选项用于设置变量的值，你可以用-DVariable=Value的形式来设置变量。对于CMake，-DVariable=Value和-D Variable=Value的效果是一样的，即在-D和变量之间有无空格并不会影响最终的结果。

然而，如果你在-D选项和其后面的变量之间有空格，那么你可能需要确保整个-D Variable=Value表达式被引号括起来（如"-D Variable=Value"），否则shell可能会将空格作为参数分隔符，这可能导致错误。

此外，对于GCC和Clang等编译器的-D选项（用于定义宏），-DVariable=Value和-D Variable=Value是不同的，后者将会导致错误。对于这些编译器，-D选项和其后面的宏定义之间不应有空格。

总结一下，对于CMake，-DVariable=Value和-D Variable=Value的效果是一样的，只要在后者中你正确地使用了引号。但是对于GCC和Clang等编译器，-D选项和其后面的宏定义之间不能有空格。

1.9 extern “C”

前面的题目中的 __cplusplus 宏，是用来识别编译器的，也就是说，将当前代码编译的时候，是否将代码作为 C++ 进行编译。如果是，则定义了 __cplusplus 宏。更多内容，这里就不详细说明了。

extern 是 C/C++ 语言中表明函数和全局变量作用范围（可见性）的关键字，该关键字告诉编译器，其声明的函数和变量可以在本模块或其它模块中使用。

注意，语句 extern int a; 仅仅是对变量的声明，其并不是在定义变量 a ，声明变量并未为 a 分配内存空间。

变量 a 在所有模块中作为一种全局变量只能被定义一次，否则会出现连接错误。

extern "C"不是让你的代码完全以C的方式编译，而是告诉链接器按照C的方式链接这些函数，从而可以让C代码调用这些函数，或者让C++代码在不考虑函数重载的情况下调用这些函数。在函数体内，你仍然可以使用C++的所有特性。

1.11 #define header & #pragma one

是的，#pragma once和传统的头文件保护宏（include guards）都是为了解决头文件多次包含的问题。但它们的工作方式和广泛性有些不同。

  1. 工作方式：
    • #pragma once：这是一个编译器指令，告诉编译器只需要包含这个文件一次，即使它在多个地方被引用。
    • #ifndef/#define：这是预处理器的指令，基于宏名称来避免文件内容的重复包含。
  2. 广泛性和可移植性：
    • #pragma once：虽然大多数现代编译器都支持这个指令，但它并不是C++标准的一部分。所以理论上它不是可移植的，尽管在实际中大多数编译器都支持它。
    • #ifndef/#define：这是C和C++预处理器的标准指令，因此它们在所有C++编译器上都是可移植的。
  3. 效率：
    • #pragma once：通常认为这个指令比传统的宏定义方法更快，因为编译器可以直接检查文件路径，而不需要处理预处理器的条件逻辑。
    • #ifndef/#define：可能会稍微慢一点，尤其是在有大量头文件时。
  4. 其他注意事项：
    • 使用#pragma once可能会在某些特殊情况下遇到问题，例如当有两个物理不同但内容完全相同的文件时。
    • 有些项目中可能会同时使用#pragma once和传统的include guards，以确保最大的可移植性和效率。

综上所述，你可以根据项目的需要和目标平台来选择使用哪种方法。如果你确定目标平台的编译器都支持#pragma once，并且想要稍微加快编译速度，那么可以使用它。否则，为了保证最大的可移植性，使用传统的include guards是更好的选择。

4. Object-Oriented Programming

4.0 Why OOP programming?

  1. 封装：OOP 允许将数据和与之相关的操作封装到一个单独的实体中，即对象。这有助于隐藏内部实现的复杂性，并提供一个清晰的界面供其他代码使用。
  2. 继承：对象能够继承其他对象的属性和方法，这使得创建基于现有对象的新对象变得容易，而无需重复代码。这也支持了代码的复用。
  3. 多态：这是一个关键的OOP概念，允许一个接口表示多种数据类型的操作。例如，你可以有多个对象，它们都有一个draw方法，但每个对象的draw方法实现都是不同的。多态使得代码更加灵活和可扩展。
  4. 抽象：对象允许程序员定义“是什么”（数据）和“能做什么”（方法），而无需关心“如何做”。这有助于减少复杂性和提高代码的可读性。
  5. 更好的模块化：对象和类为代码提供了自然的组织结构，使得功能和数据更加模块化，易于维护。
  6. 设计模式：OOP引入了许多设计模式，这些模式为常见的编程问题提供了解决方案。这使得软件设计更加稳定和可靠。

对象声明的时候，有两种不同的形式：

在堆上分配内存的优点是您可以手动控制内存的生命周期，并且可以动态地分配和释放内存。

  1. 内存管理：栈上的变量由编译器自动管理内存，它们的内存分配和释放是自动进行的。而堆上的变量需要手动分配和释放内存，通过 new 进行分配，通过 delete 进行释放。
  2. 生命周期：栈上的变量的生命周期与其所在的作用域相对应。当变量超出作用域时，其内存会自动释放。而堆上的变量的生命周期由程序员控制，必须手动释放内存，否则可能会导致内存泄漏。
  3. 内存分配方式：栈上的内存分配速度较快，仅仅是移动栈指针。而堆上的内存分配需要在堆空间中寻找可用的内存块，并进行分配，速度相对较慢。
  4. 变量的大小：栈空间有限，通常较小，因此栈上的变量大小有限制。而堆空间相对较大，可以分配更大的变量。

9.0.1 容器分类

  • 基于连续内存的：在移动元素（删除/添加/移动)时会将整个内存块的元素移动，比较影响效率。

    1. std::list: 它是一个双向链表，可以在任何位置进行快速的插入和删除操作，但不支持随机访问。

当且仅当新的vector大小超过当前vector容量时，会重新自动分配新的存储空间。

对于vector来说，和数组最大的区别之一，就是不需要在初始化的时候声明vector的大小。如果初始化的时候没有指明vector的大小，那么会根据实际的使用情况，在内存中为vector分配的大小分别2 -> 4 -> 8 -> 16 … 。

重新分配空间，并不是接着原来的地址进行分配，而是会将前面的所有元素统一移动到一个新的地方，再重新进行内存分配，这样就会导致很大的时间开销。

为了避免重新分配内存带来的问题，vector提供了reserve函数。

reserve的作用是更改vector的容量（capacity），使vector至少可以容纳n个元素。

如果n大于vector当前的容量，reserve会对vector进行扩容。其他情况下都不会重新分配vector的存储空间。

9.1.3 reserve or resize

void resize(size_type n, value_type val = value_type());

  • 如果n<当前容器的size，则将元素减少到前n个，移除多余的元素(并销毁）
  • 如果n>当前容器的size，则在容器中追加元素，如果val指定了，则追加的元素为val的拷贝，否则，默认初始化
  • 如果n>当前容器容量，内存会自动重新分配

void reserve(size_type n)

  • 如果n>容器的当前capacity，该函数会使得容器重新分配内存使capacity达到n
  • 任何其他情况，该函数调用不会导致内存重新分配，并且容器的capacity不会改变
  • 该函数不会影响向量的size而且不会改变任何元素

C++字符串（std::string）和C字符串（以null结尾的字符数组）在表示和操作上有一些不同之处。

  1. 表示方式：C++字符串是一个类对象，由C++标准库提供，使用std::string类型表示。它内部封装了一个动态分配的字符数组，并提供了丰富的成员函数用于字符串操作。而C字符串是一个以null结尾的字符数组，使用字符指针（char*）来表示，没有对应的类型。
  2. 存储方式：C++字符串自动处理内存分配和释放，可以动态调整字符串的长度。它使用动态内存分配（如new和delete）或者使用标准库提供的内存管理机制。而C字符串需要手动管理内存，需要使用函数如malloc和free进行内存的分配和释放。
  3. 字符串长度：C++字符串对象（std::string）包含一个成员变量来记录字符串的长度，可以通过成员函数length()或size()来获取。而C字符串没有内置的长度信息，需要使用函数如strlen来计算字符串的长度。
  4. 字符串操作：C++字符串提供了丰富的成员函数和操作符重载，用于字符串的连接、查找、截取、替换等常见操作。而C字符串需要使用C标准库中的函数（如strcpy、strcat、strcmp等）来执行类似的操作。

C++字符串相对于C字符串具有更高的安全性和方便性，可以动态调整长度、自动处理内存分配和释放，提供了更多的操作函数和运算符重载。因此，在使用C++编程时，推荐使用std::string来处理字符串，特别是涉及到字符串操作和动态长度的场景。但是在与C语言代码交互、调用C库函数等情况下，可能需要进行C字符串和C++字符串之间的转换。

12.5 传入pointer与传入shared_ptr的区别

在开发多线程程序中，无意中发现使用raw pointer 与使用 shared pointer 的异同。

shared_ptr更安全，并且建议在传入线程参数的时候，都使用shared_ptr ，具体原因如下：

  1. 内存管理：shared_ptr 确保自动管理内存。当最后一个 shared_ptr（拥有指向特定对象的引用）被销毁或被重置时，它所指向的对象会被自动删除。这避免了可能因忘记释放内存而导致的内存泄漏。
  2. 引用计数：shared_ptr 使用引用计数机制来追踪有多少个 shared_ptr 实例指向同一个对象。这个引用计数是线程安全的，这意味着在多线程环境中，你不需要额外的同步机制来更新这个计数。
  3. 线程安全：虽然 shared_ptr 本身的引用计数操作是线程安全的，但是你需要确保不同时对其所指向的对象进行写入操作，除非你使用了适当的同步机制。
  4. 所有权语义清晰：使用 shared_ptr 能够清晰地表达对象的所有权和生命周期。当你看到一个函数接受 shared_ptr 作为参数或返回 shared_ptr 时，你可以清晰地知道对象的所有权如何传递和管理。
