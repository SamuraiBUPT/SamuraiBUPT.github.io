1.3 package.json

package.json 是一个标准的文件，用于 Node.js 和 npm 项目。它描述了项目的各种元数据、依赖、脚本和其他配置。对于任何使用 npm 或 yarn 作为包管理器的项目来说，这个文件都是核心的。

  1. 基础信息:
    • name: 项目的名称。
    • version: 项目的版本号。
    • description: 项目的简短描述。
  2. 联系和作者信息:
    • author: 项目作者的信息。
    • contributors: 项目的贡献者列表。
    • license: 项目使用的许可证。
  3. 项目配置:
    • scripts: 用于存储和运行常用脚本的命令。例如：start, test, build 等。
    • main: 指定当此包被引用时的主入口文件。
    • engines: 指定项目兼容的 Node.js 或 npm 版本。
  4. 依赖管理:
    • dependencies: 列出项目运行时所需的依赖。
    • devDependencies: 列出开发时所需的依赖，例如测试工具、构建工具等。这些依赖在生产环境中通常不需要。
    • peerDependencies: 列出项目预期与之共同存在的依赖。这主要用于插件或库，它们希望主项目有某些依赖但不希望自己直接包含这些依赖。
  5. 其他配置和元数据:
    • private: 如果设置为 true，则防止该包被意外发布到 npm。
    • repository: 项目的代码存储库信息。
    • keywords: 关键词数组，有助于在 npm 中搜索。
    • browserslist: 用于配置目标浏览器，常与 Autoprefixer、Babel 等工具一起使用。

选项式 API:

  • 选项式 API 是 Vue 的传统方式来组织和定义组件。这种方式主要基于“选项”的思路，你会将组件的逻辑按照其功能类型（如 data、methods、computed、watch、props 等）组织到不同的选项中。

组合式 API:

  • 组合式 API 是 Vue 3 新引入的，提供了一种更灵活的方式来组织和复用组件逻辑。它的核心是 ref、reactive、computed 和 watch 等函数，允许你在单个 setup 函数内组合和整合逻辑。
  • 与选项式 API 相比，组合式 API 更容易使相关功能逻辑保持在一起，特别是当组件逻辑变得复杂时。

响应式:

  • 响应式是 Vue 核心的一部分，它描述了当数据变化时如何自动更新 UI 的过程。
  • 在 Vue 中，你可以使用 ref 和 reactive 创建响应式的数据源。这意味着当这些数据发生变化时，任何依赖这些数据的 UI 部分都会自动更新。
  • ref 用于基本类型数据的响应式，而 reactive 用于创建响应式对象。

全局注册的问题：

  1. 全局注册，但并没有被使用的组件无法在生产打包时被自动移除 (也叫“tree-shaking”)。如果你全局注册了一个组件，即使它并没有被实际使用，它仍然会出现在打包后的 JS 文件中。
  2. 全局注册在大型项目中使项目的依赖关系变得不那么明确。在父组件中使用子组件时，不太容易定位子组件的实现。和使用过多的全局变量一样，这可能会影响应用长期的可维护性。

相比之下，局部注册的组件需要在使用它的父组件中显式导入，并且只能在该父组件中使用。它的优点是使组件之间的依赖关系更加明确，并且对 tree-shaking 更加友好。

解释：

  1. '/api': 当你在应用中发起一个以 /api 开头的请求时，它会被代理到在 target 选项中指定的URL。

    例如，假设你的开发服务器运行在 http://localhost:8080，并且你发起了一个请求到 http://localhost:8080/api/user。由于代理配置，这个请求实际上会转发到 <url>/api/user。

  3. changeOrigin: true: 当为 true 时，代理服务器会修改请求头的 host 为 target 的主机名。例如，如果 target 是 http://example.com，那么请求头的 host 会被设置为 example.com。这对于某些检查主机头的后端服务器是必要的。

所以，设置好<target url>，即可解决跨域匹配的问题。

5. 浏览器缓存与用户数据

这一部分，主要是在做“用户权限验证”这一块的时候，会涉及到。以下是GPT的回答：

登录验证（通常被称为身份验证）是许多应用程序中的关键功能。这种功能确保了只有经过验证的用户才能访问特定资源。以下是前端身份验证的一般流程和概念：

  1. 用户提交凭据：用户在登录表单中输入用户名和密码，然后点击登录。
  2. 后端验证：前端将这些凭据发送到后端服务器，后端进行验证。如果凭据正确，后端会生成一个 token（通常是 JSON Web Token，即 JWT）。
  3. 前端存储 Token：一旦后端验证用户并返回 token，前端会将 token 存储在某个地方，通常是：
    • Cookie：是由浏览器存储的小型数据片段，可以在后续请求中发送回服务器。
    • LocalStorage 或 SessionStorage：Web 浏览器提供的用于存储键值对的 Web API。它们不会过期（除非用户或代码删除它们），但与 Cookie 不同，它们不会在每个请求中自动发送回服务器。
  4. 后续请求的身份验证：每当前端发起到受保护资源的请求时，它都会在请求头中附带该 token。后端会检查此 token，如果 token 有效，则允许访问。
  5. 路由守卫：在前端，特别是在像 Vue 或 React 这样的框架中，你可以使用“路由守卫”或“路由拦截器”来检查用户是否登录。如果用户没有登录但试图访问受保护的页面，它会将用户重定向到登录页面。
  6. 退出登录：用户决定退出登录时，前端将删除存储的 token 并可能通知后端使其过期。
  7. Token 刷新：为了安全起见，token 通常有一个较短的有效期。一旦 token 过期，用户需要重新登录以获取新的 token。然而，也存在“刷新 token”的概念，它允许应用程序在不要求用户重新登录的情况下更新过期的 token。
  8. 跨站请求伪造（CSRF）：当使用 cookies 存储 token 时，需要考虑防范 CSRF 攻击。简而言之，CSRF 是攻击者利用用户的登录状态进行恶意请求的一种攻击方法。为了防范它，除了 cookie 外，通常还会发送一个伪随机 token，并在每个请求中验证它。

总之，身份验证流程涉及前端和后端的紧密合作。前端负责收集凭据、存储 token、附带 token 发起请求和管理用户登录状态，而后端则负责验证用户凭据、生成 token 和验证每个受保护请求的 token。
