2. 对指针本体的剖析

2.1 指针的几种基本用法

指针有很多种用法，指针本身的作用，就是指向一块内存空间，如果指针不指向任何地址，那么这个指针就是无用的指针。

基于上面这句描述以及其理解，笔者对常见指针的用法，加以分类。

1. 通过指针，掌握内存区块

这是最常见的，也是最重要的功能。这个 内存区块 的概念，是非常抽象而博大的。

指针指向的内存资源，可以是一块栈内存，也可以是一块堆内存（通过new分配），还可以是一个被加载的资源文件。（比如深度学习模型的权重文件，动辄上百M），这些庞杂的资源，都需要通过指针进行内存管理。

3.1 指针大小

无论是哪种类型的指针，默认大小都是4 bytes (32位系统)，因为一个指针存储的地址是0xffffffff.

**sizeof()**的单位是字节（byte），例如sizeof(int)返回4。

而在64位系统上面，一个指针的大小是 8 bytes。

我们大多数人用的电脑都是64位系统，理所当然指针的大小是8字节。

指针可以直接与整数进行运算。

需要注意的是，void*指针类型不可以直接进行指针运算。如果需要对指针运算取偏移，那么必须要把这个指针转换为其他类型，才可以进行运算。

4. 智能指针

4.1 unique_ptr

头文件：memory

当涉及到内存资源管理时，"内存资源所有权"是一个重要的概念。它指的是程序中某个实体（通常是指针）对于分配的内存资源的所有权，即负责管理该内存资源的生命周期。

在传统的C++开发中，我们使用new和delete关键字手动分配和释放内存。这意味着我们需要手动跟踪每个内存资源的所有权，并确保在不再使用该资源时正确释放它，以避免内存泄漏和悬挂指针等问题。这种手动管理内存的方式容易出错，尤其是在复杂的代码中。

为了简化内存管理并提高代码的安全性和可维护性，C++11引入了智能指针，其中包括std::unique_ptr。std::unique_ptr是一个独占所有权的智能指针，它负责管理分配的内存资源，并在不再需要时自动释放它。通过使用std::unique_ptr，我们可以将内存资源的所有权委托给一个特定的指针，从而消除了手动跟踪和释放内存的责任。

std::unique_ptr 实现了独享所有权的语义。一个非空的 std::unique_ptr 总是拥有它所指向的资源。转移一个std::unique_ptr 将会把所有权也从源指针转移给目标指针（源指针被置空）。拷贝一个std::unique_ptr 将不被允许，因为如果你拷贝一个 std::unique_ptr ,那么拷贝结束后，这两个std::unique_ptr 都会指向相同的资源，它们都认为自己拥有这块资源（所以都会企图释放）。因此std::unique_ptr 是一个仅能移动（move_only）的类型。当指针析构时，它所拥有的资源也被销毁。默认情况下，资源的析构是伴随着调用std::unique_ptr 内部的原始指针的 delete 操作的。

4. Why Smart Pointer?

  1. 所有权语义：std::unique_ptr通过其名字明确表示其管理的对象的所有权是唯一的。这可以在阅读代码时提供有关资源管理的有用信息。也就是能保证多个shared_ptr共享某一个对象资源的所有权 。
  2. 引用计数：std::shared_ptr允许多个智能指针共享同一个对象的所有权，它使用引用计数来确保只有最后一个对该对象的引用被销毁时，对象才被删除。
  3. 异常安全：如果函数因异常而提前退出，所有在该函数中构造的对象都会被析构。如果你使用原始指针，可能会因为异常而忘记释放内存，但如果使用智能指针，内存会被自动管理。
  4. 可自定义的删除器：你可以为std::unique_ptr和std::shared_ptr提供自定义的删除器，以处理特殊的资源释放需求。
  5. 与STL容器兼容：智能指针设计为可以与STL容器一起使用，这比原始指针更加安全。
  6. 提高代码的可读性：通过使用智能指针，你可以明确地在代码中表示资源所有权和生命周期，使代码的意图更加清晰。

6. use_count

use_count (使用计数)：表示有多少个 shared_ptr 实例当前正在共享同一资源（或指向同一个对象）。当你复制一个 shared_ptr 或使一个新的 shared_ptr 指向一个已经被其他 shared_ptr 所指向的对象时，use_count 会增加。每当一个 shared_ptr 被销毁或被重置，use_count 会减少。当 use_count 降到0时，指向的对象会被自动删除。

use_count 只在以下情况改变：

  1. 当你复制一个 std::shared_ptr，如 std::shared_ptr<T> ptr2 = ptr1;。
  2. 当你用一个已存在的 std::shared_ptr 初始化另一个 std::shared_ptr。
  3. 当一个 std::shared_ptr 被销毁，如超出作用域或被重置。
  4. 当你将一个 std::shared_ptr 分配给另一个已存在的 std::shared_ptr。

简单地使用一个指向 std::shared_ptr 的指针，不会导致 use_count 的改变。比如函数传参，传递一个指向智能指针的指针进去。

智能指针，尤其是shared_ptr的出现，解决了资源控制权的问题，同时也减少了野指针出现的可能。

4.4 函数传参

在函数传参的时候，有时候甚至能看到“智能指针的引用”或者“智能指针的指针”。

这是因为，智能指针本身就是一个对象。如果按照值传递，就会拷贝这个对象一次。

  1. 以值传递 (pass by value): 传入 std::shared_ptr 的拷贝。这会增加共享对象的引用计数。函数内部对拷贝的 shared_ptr 进行的操作不会影响原始的 shared_ptr，但是会影响它所管理的对象（除非你修改拷贝的 shared_ptr 使其指向另一个对象）。

  2. 以引用传递 (pass by reference): 传入对 std::shared_ptr 的引用。这不会增加引用计数。函数内部对 shared_ptr 的任何操作都会影响到原始的 shared_ptr。

  3. 以常量引用传递 (pass by const reference): 传入对 std::shared_ptr 的常量引用。这不会增加引用计数。函数可以读取 shared_ptr 但不能修改它。

5. const修饰指针

const int* a 和 const int* b 是指向常量整型的指针。这意味着它们可以指向一个 int 类型的常量，但不能通过它们修改所指向的值。

也就是说，可以修改指针的赋值，但是不能通过指针，去修改上层内存的值。

更进一步：可以赋值，但是只能读取值，无法通过指针取修改。

在指针前面声明const，仅仅代表你不会通过这个指针去修改原来内存上面的值，而不代表不去给这个指针赋值。

9. void * 的应用

void*，即void指针，是C和C++编程中一个特殊的指针类型，它可以指向任何数据类型的地址，包括函数。使用void指针有一些特定的目的和考虑：

  1. 通用性：void指针在某些情况下被用作通用数据类型。例如，在C的标准库函数（如malloc和free）中，malloc返回一个void指针，这意味着它可以被分配给任何类型的指针。
  2. 回调函数与用户数据：在很多API或库中，当提供一个回调函数时，通常还允许用户提供一个void*类型的参数，该参数可以指向用户定义的数据。这使得在回调函数被调用时可以访问这些数据。这种设计模式增加了API的灵活性，使其可以处理各种数据类型。
  3. 抽象与封装：有时，库或API希望隐藏或封装其内部数据结构的具体细节。通过仅公开void指针接口，它可以防止用户直接访问或修改内部数据。
  4. 类型转换：在某些复杂的情况下，你可能需要进行各种类型的转换。使用void指针可以作为一个中间转换步骤，然后再转换为目标类型。
  5. 与其他语言的兼容性：在与其他编程语言的接口中，有时使用void指针来表示一个不确定类型的引用。

尽管void*提供了很大的灵活性，但它也有一些限制：

  • 类型安全：由于void指针不提供关于其所指向内容的任何类型信息，所以编译器不能提供类型检查。这可能会导致运行时错误，如果开发者没有正确地处理void指针。
  • 指针运算：不能直接在void指针上执行指针运算，因为系统不知道应该增加或减少多少来移动到下一个元素。

总的来说，void指针是一个强大且灵活的工具，但也需要谨慎使用。在现代C++编程中，模板和其他类型安全的机制往往是更好的选择，但在C和某些特定的C++场景中，void指针仍然很有用。

STL容器的 .at() 方法返回的确是一个引用。你所给出的例子中使用了 std::deque<uthread_tcb *>，该容器中的 .at() 方法返回的是对 uthread_tcb * 的引用。

总结：这两者不同。前者得到的是容器元素的引用，后者得到的是容器元素的副本。
